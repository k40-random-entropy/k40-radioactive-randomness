Purpose

This document describes the secure delivery process for measurement data and random‑number exports generated by this project. It ensures confidentiality, integrity, and authenticity when files are transmitted to customers.

Integrity: Digital Signatures

All exported files are digitally signed using minisign.
Customers can verify authenticity with the following command:<br>
minisign -Vm file.json -p publickey.txt<br>
This guarantees that the file originates from the project owner, has not been modified, and is complete.

Confidentiality: AES‑256 Encryption

If a customer requires confidentiality, files are encrypted locally using AES‑256 before transmission.
Example encrypted file: <br>export_data_zip.aes<br>
The encrypted file can safely be transmitted over untrusted channels such as email.

Identity Verification of the Customer

To prevent man‑in‑the‑middle attacks, the customer must initiate contact via WhatsApp.
The customer sends a short WhatsApp message to the project owner’s verified number.
Only after this step is the customer’s identity considered verified.
This prevents attackers from sending fake phone numbers via email.

Password Transmission (End‑to‑End Encrypted)

The AES‑256 password is sent only as a reply to the WhatsApp message initiated by the customer.
This ensures end‑to‑end encryption, verified identity, and prevents password exposure via email.

File Transmission

After identity verification:<br>
– the encrypted file is sent via email<br>
– the password is sent via WhatsApp<br>
– the minisign signature and public key are provided for integrity checking

Optional: Customer Public Key

If the customer provides a PGP key, age key, or minisign public key, files can be encrypted directly for that key.
This enables full end‑to‑end encryption without passwords.

SICHERHEITSRICHTLINIE (Deutsche Version)

Zweck

Dieses Dokument beschreibt das sichere Übertragungsverfahren für Messdaten und Zufallszahl‑Exporte dieses Projekts. Es stellt Vertraulichkeit, Integrität und Authentizität sicher.

Integrität: Digitale Signaturen

Alle exportierten Dateien werden mit minisign digital signiert.
Der Kunde kann die Echtheit prüfen mit:<br>
minisign -Vm Datei.json -p publickey.txt<br>
Dies garantiert, dass die Datei vom Projektinhaber stammt, nicht verändert wurde und vollständig ist.

Vertraulichkeit: AES‑256‑Verschlüsselung

Falls der Kunde Vertraulichkeit benötigt, werden Dateien lokal mit AES‑256 verschlüsselt.
Beispiel:<br> export_data_zip.aes<br>
Die verschlüsselte Datei kann gefahrlos über unsichere Kanäle wie E‑Mail übertragen werden.

Identitätsprüfung des Kunden

Um Man‑in‑the‑Middle‑Angriffe zu verhindern, muss der Kunde zuerst den Projektinhaber per WhatsApp kontaktieren.
Der Kunde sendet eine kurze WhatsApp‑Nachricht an die verifizierte Nummer des Projektinhabers.
Erst danach gilt die Telefonnummer als authentisch.
Dies verhindert, dass Angreifer gefälschte Telefonnummern per E‑Mail einschleusen.

Passwortübermittlung (Ende‑zu‑Ende‑verschlüsselt)

Das AES‑256‑Passwort wird ausschließlich als Antwort auf die WhatsApp‑Nachricht des Kunden gesendet.
Damit ist gewährleistet: Ende‑zu‑Ende‑Verschlüsselung, verifizierte Identität, kein Passwort in E‑Mails.

Dateiübertragung

Nach erfolgreicher Identitätsprüfung:<br>
– die verschlüsselte Datei wird per E‑Mail gesendet<br>
– das Passwort per WhatsApp<br>
– minisign‑Signatur und Public Key werden zur Integritätsprüfung bereitgestellt

Optional: Öffentlicher Schlüssel des Kunden

Falls der Kunde einen PGP‑, age‑ oder minisign‑Public‑Key bereitstellt, kann die Datei direkt für diesen Schlüssel verschlüsselt werden.
Dies ermöglicht vollständige Ende‑zu‑Ende‑Verschlüsselung ohne Passwort.

Öffentlicher Schlüssel zur Signaturprüfung
